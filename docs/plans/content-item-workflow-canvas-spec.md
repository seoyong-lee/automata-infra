# 제작 아이템 상세 작업 캔버스 UX 스펙

본 문서는 Admin Web의 제작 아이템 상세 화면을 **탭 전환형 상세**에서 **단계형 작업 캔버스**로 재구성하기 위한 UX 기준 문서다.

관련 문서:

- [`non-technical-admin-publish-flow-spec.md`](./non-technical-admin-publish-flow-spec.md)
- [`admin-client-revision-workbench.md`](./admin-client-revision-workbench.md)

---

## 0. 한 줄 결론

제작 아이템 상세는 앞으로 아래 구조로 고정한다.

- 상단 = 항상 보이는 워크플로 상태 바
- 하단 = 현재 선택한 단계의 상세 작업 화면

즉, 사용자는 한 화면 안에서 항상 아래 두 가지를 동시에 알아야 한다.

1. 지금 전체 흐름 중 어디까지 왔는가
2. 지금 이 단계에서 무엇을 해야 하는가

---

## 1. 왜 이 구조가 필요한가

현재 상세 화면의 핵심 문제는 아래 세 가지다.

### 1.1 흐름이 화면과 탭으로 지나치게 분산된다

사용자는 제작 아이템을 하나 끝까지 밀고 나가는 작업을 기대하는데, 현재는 탭과 별도 화면을 오가며 맥락이 자주 끊긴다.

### 1.2 전제 조건이 숨겨져 있다

채널 연결, 매체 연결, 검수 완료, 발행 문구 작성, 큐 적재 같은 선행 조건이 명확한 순서로 드러나지 않는다.

### 1.3 시스템 구조가 사용자 흐름보다 앞에 나온다

비개발자 운영자에게 중요한 것은 파이프라인 내부 구조가 아니라:

- 지금 단계
- 다음 행동
- 막힌 이유
- 완료 여부

따라서 상세 화면은 여러 도구의 집합이 아니라 **단일 작업 공간**이어야 한다.

---

## 2. 최종 UX 방향

기존의 "탭 여러 개 + 별도 화면 이동 중심" 대신, 제작 아이템 상세를 **하나의 작업 캔버스**로 바꾼다.

핵심 원칙:

- 전체 흐름은 상단에서 항상 보인다
- 실제 작업은 하단 단계 패널에서만 한다
- 현재 단계의 메인 CTA는 하나만 둔다
- 페이지 이동보다 패널 전환이 기본이다
- 외부 화면 이동은 보조 CTA로만 둔다

이 구조는 비개발자 운영자가 아이템 하나를 끝까지 처리하는 데 최적화된 형태다.

---

## 3. 정보 구조

제작 아이템 상세 페이지는 아래 3층으로 구성한다.

### 3.1 A. 상단 헤더

역할:

- 아이템의 정체성과 현재 상태를 요약한다
- 지금 가장 중요한 다음 행동 하나를 보여준다

포함 항목:

- 아이템 제목
- 현재 단계
- 마지막 수정 시각
- 연결된 채널 수
- 발행 준비 상태 배지
- 우측 메인 CTA 1개

메인 CTA 규칙:

- 현재 단계에 따라 하나만 노출
- 보조 CTA는 최대 2개
- 사용자가 다음 행동을 고민하지 않게 해야 함

예시:

- `검수 계속`
- `발행 문구 작성`
- `출고 큐에 추가`
- `예약·발행으로 이동`

### 3.2 B. 상단 고정 워크플로 바

역할:

- 전체 진행 상황을 시각적으로 보여준다
- 사용자가 현재 위치를 잃지 않게 한다
- 하단 작업 패널을 어떤 단계로 볼지 선택하게 한다

이 바는 **편집 가능한 노드 그래프가 아니라 상태형 스테퍼**여야 한다.

권장 단계:

1. 아이디어
2. 스크립트
3. 에셋 생성
4. 검수
5. 발행 문구
6. 출고 대기
7. 예약·발행
8. 결과

노드 상태:

- 완료
- 진행 중
- 필요
- 막힘
- 선택됨

노드 클릭 시 동작:

- 페이지 전체 이동이 아니라 하단 상세 패널 전환
- 외부 화면이 꼭 필요한 경우에만 보조 CTA 사용

### 3.3 C. 준비 상태 체크리스트

워크플로 바 옆 또는 바로 아래에 **준비 상태 칩**을 고정한다.

필수 항목:

- 채널 연결 완료
- 매체 연결 완료
- 검수 완료
- 발행 문구 작성 완료
- 출고 큐 적재 완료

각 칩 상태:

- 완료
- 필요
- 막힘

각 칩 클릭 시:

- 관련 단계 패널로 이동

### 3.4 D. 하단 상세 작업 패널

역할:

- 현재 선택한 단계의 상세 작업을 수행한다

상단과 하단의 역할 분리:

- 상단: 어디까지 왔는지
- 하단: 지금 여기서 뭘 하는지

하단 패널은 단계별로 교체되며, 항상 아래 구조를 기본으로 권장한다.

- 좌측: 주요 작업 영역
- 우측: 상태 요약, 다음 단계, 경고/오류, 관련 링크

---

## 4. 단계별 상세 패널 구성

### 4.1 아이디어 패널

목적:

- 이 아이템이 무엇인지 빠르게 이해

구성:

- 한 줄 설명
- 소스/주제/기획 메모
- 생성 이력 요약
- 관련 산출물 수

CTA:

- `스크립트 보기`
- `새 버전 만들기`

원칙:

- 편집보다는 컨텍스트 확인 중심

### 4.2 스크립트 패널

목적:

- 대본/카피/구성 확정

구성:

- 최신 스크립트 미리보기
- 버전 목록
- 승인 상태
- 수정/재생성 버튼

CTA:

- `스크립트 승인`
- `에셋 생성으로 진행`

### 4.3 에셋 생성 패널

목적:

- 이미지/음성/영상 등 결과물 확인

구성:

- 필터 또는 서브탭: 이미지 / 음성 / 영상 / 썸네일
- 최신 결과물 카드
- 실패한 작업 카드
- 재시도 버튼

원칙:

- 기본 화면에 내부 식별자를 올리지 않는다
- `assetId` 같은 값은 고급 정보로만 노출

CTA:

- `대표본 선택`
- `검수로 진행`

### 4.4 검수 패널

목적:

- 사람이 최종 확인하고 승인/반려

구성:

- 검수 체크리스트
- 승인 / 반려 버튼
- 반려 사유
- 변경 요청 이력

원칙:

- 1차 검수는 아이템 상세 안에서 처리
- 전역 리뷰 화면 이동은 보조 CTA로만 제공

CTA:

- `검수 승인`
- `수정 요청`
- 보조: `전체 검수 큐에서 보기`

### 4.5 발행 문구 패널

목적:

- 플랫폼용 제목/설명/해시태그/썸네일 결정

구성:

- 플랫폼별 문구 탭
- 제목 / 설명 / 태그
- 썸네일 선택
- 미리보기

원칙:

- `썸네일 에셋 ID` 입력 제거
- 실제 사용자 언어로 재작성

CTA:

- `발행 문구 저장`
- `출고 대기 추가`

### 4.6 출고 대기 패널

목적:

- 이 아이템이 발행 준비 완료 상태인지 확인

구성:

- 준비 체크리스트
- 연결된 채널
- 발행 대상
- 큐 적재 여부
- 큐 추가 시각

필수 체크리스트:

- 채널 연결 완료
- 매체 연결 완료
- 검수 완료
- 발행 문구 작성 완료
- 출고 큐 적재 완료

CTA:

- `출고 큐에 추가`
- `예약·발행으로 이동`

### 4.7 예약·발행 패널

목적:

- 실제 언제, 어디에 올릴지 결정

구성:

- 채널별 또는 플랫폼별 발행 카드
- 플랫폼별 예약 시간
- 즉시 발행 가능 상태
- 예약 저장 / 예약 해제 / 지금 발행

중요 판단:

- 아이템 상세 안에서는 "이 아이템 하나 기준 예약/발행"
- 채널 예약·발행 페이지는 "채널 전체 운영 보드"

즉, 두 화면은 대체 관계가 아니라 역할 분리 관계다.

### 4.8 결과 패널

목적:

- 발행 성공/실패와 외부 링크 확인

구성:

- 마지막 실행 결과
- 성공/실패 상태
- 실패 원인
- 외부 링크
- 재시도 버튼

오류 메시지 원칙:

- 무엇이 실패했는지
- 왜 실패했는지
- 다음에 무엇을 해야 하는지

## 5. 핵심 인터랙션 규칙

### 5.1 상단 워크플로는 항상 보여야 한다

- 스크롤과 무관하게 현재 위치를 잃지 않아야 한다
- 헤더 아래 sticky 고정
- 가로 스크롤 허용
- 좁은 화면에서는 축약형 지원

### 5.2 노드 클릭은 "이동"보다 "전환"이 기본이다

- 라우팅보다 하단 패널 상태 전환 우선
- 페이지가 바뀌면 비개발자는 맥락을 잃기 쉽다

### 5.3 상단은 요약, 하단은 실행

상단 워크플로 노드에는 아래 정도만 둔다.

- 상태
- 짧은 제목
- 경고 점 또는 상태 배지

실제 수정/저장/실행 버튼은 하단 패널에 둔다.

### 5.4 메인 CTA는 언제나 하나다

각 시점에 사용자가 가장 먼저 눌러야 하는 버튼은 하나만 노출한다.

예:

- 검수 전: `검수 계속`
- 검수 후 문구 미작성: `발행 문구 작성`
- 문구 저장 후 큐 미적재: `출고 큐에 추가`
- 큐 적재 후: `예약·발행으로 이동`

### 5.5 내부 용어는 기본 UI에 올리지 않는다

숨길 값:

- jobId
- assetId
- oauthAccountId
- uploadVideoId
- platformConnectionId

필요하면 `고급 정보` 또는 `기술 정보` 접힘 영역으로 이동한다.

## 6. 컴포넌트 구조 제안

상세 화면은 `Overview + WorkflowDetail` 2단 구조로 개편한다.

권장 상위 구조:

- `ContentJobHeader`
- `ContentJobWorkflowBar`
- `ContentJobReadinessChecklist`
- `ContentJobWorkflowPanel`

패널 내부 단계 컴포넌트:

- `IdeaStagePanel`
- `ScriptStagePanel`
- `AssetsStagePanel`
- `ReviewStagePanel`
- `PublishDraftStagePanel`
- `QueueStagePanel`
- `ScheduleStagePanel`
- `ResultStagePanel`

권장 레이아웃:

```text
[아이템 제목] [현재 단계 배지] [메인 CTA]

[워크플로 바]

[준비 상태 칩]

--------------------------------------------------
[선택된 단계 상세 패널]

좌측
- 주요 작업 영역
- 미리보기 / 입력 / 상태 / 이력

우측
- 다음 단계
- 현재 상태 요약
- 관련 링크
- 실패/주의 메시지
--------------------------------------------------
```

## 7. 상태 모델 제안

기존 탭 모델 대신, 상세 화면 전체가 동일한 워크플로 상태 모델을 바라보도록 한다.

```ts
type WorkflowStageKey =
  | "idea"
  | "script"
  | "assets"
  | "review"
  | "publishDraft"
  | "queue"
  | "schedule"
  | "result";

type WorkflowStageStatus = "done" | "current" | "blocked" | "todo";

type WorkflowStage = {
  key: WorkflowStageKey;
  label: string;
  status: WorkflowStageStatus;
  description?: string;
  badge?: string;
  actionLabel?: string;
};
```

이 모델을 아래가 함께 사용해야 한다.

- 상단 워크플로 바
- 준비 체크리스트
- 메인 CTA 계산
- 하단 패널 선택 상태

## 8. 기존 화면과의 관계

### 8.1 기존 탭 구조

현재의 `detail-workspace-tabs.ts` 기반 구조는 "탭 나열"이 아니라 "단계 모델" 중심으로 치환하는 것이 바람직하다.

즉, 상세 IA는 다음처럼 바뀐다.

- 탭 목록 중심
- 단계 상태 중심

### 8.2 예약·발행 페이지와의 관계

예약·발행은 상세 안에도 패널이 있고, 채널 전역 페이지도 유지한다.

역할 분리:

- 아이템 상세 안: 이 아이템 하나 기준 실행
- 채널 예약·발행 페이지: 채널 전체 운영 보드

### 8.3 전역 리뷰 페이지와의 관계

검수는 1차적으로 아이템 상세 안에서 수행하고, 전역 리뷰 화면은 예외 처리나 전체 운영 관제용으로 둔다.

즉:

- 기본 = 상세 안에서 처리
- 보조 = 전역 검수 큐 이동

### 3차

목표:

- 재시도

## 10. 완료 판단 기준

비개발자 운영자가 설명 없이 아래를 수행할 수 있어야 한다.

1. 지금 어느 단계인지 안다
2. 왜 막혔는지 안다
3. 다음에 눌러야 할 버튼이 무엇인지 안다
4. 검수, 발행 문구, 출고 대기, 예약·발행을 한 화면 안에서 이어서 처리할 수 있다
5. 전체 채널 운영이 필요할 때만 채널 예약·발행 페이지로 이동한다

완료 조건:

- 상세 화면만 보고도 현재 상태와 다음 행동을 추론할 수 있어야 함
- 페이지 이동이 줄어들어야 함
- 내부 용어 없이도 핵심 행동이 가능해야 함

## 11. 최종 판단

이 구조는 단순히 "탭을 예쁘게 바꾸는 일"이 아니다.

핵심은 상세 화면을 시스템 정보 모음이 아니라 **운영자용 단계형 작업 캔버스**로 바꾸는 것이다.

최종 방향은 아래로 고정한다.

- 상단 고정 워크플로: 항상 현재 위치와 전체 흐름 표시
- 준비 체크리스트: 왜 막혔는지 즉시 설명
- 하단 단계 상세 패널: 지금 선택한 단계의 실제 작업
- 메인 CTA 하나: 다음 할 일 명확화
- 예약·발행은 인라인 패널 + 채널 전역 페이지 병행

이 기준은 비개발자 운영자 UX, 문서 기준, 그리고 현재 코드베이스의 점진적 개편 방향을 모두 만족하는 가장 현실적인 안이다.
