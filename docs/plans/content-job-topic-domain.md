# 콘텐츠 · 잡 · 토픽 도메인 정리

Admin·백엔드에서 혼용되기 쉬운 **콘텐츠(Content)**, **잡(Job)**, **토픽 시드/플랜(Topic seed / plan)**의 관계와, 향후 **아이디어 선행 → 채널(콘텐츠) 부착** 모델로 확장할 때의 기준을 한곳에 고정한다.

관련 UI·IA 상위 문서: [`admin-job-authoring.md`](./admin-job-authoring.md).

---

## 1. 현재 구현 기준(as-is) 관계

### 계층

```
콘텐츠 (Content, contentId)     … 카탈로그 단위. 유튜브 채널과 맞춘 “발행/운영 단위”
    └── 잡 (Job, jobId)         … 한 편(또는 한 실행)의 파이프라인 인스턴스. contentId로 콘텐츠에 연결(레거시 잡은 비어 있을 수 있음)
            └── 산출물 (S3 + 잡 메타) … 시드, 브리프, 토픽 플랜, 씬 JSON 등은 모두 “이 잡”에 귀속
```

### 핵심 규칙

- **토픽 플랜의 저장·식별 단위는 잡이다.**  
  예: S3 경로 `topics/{jobId}/topic.json`, 잡 메타의 `topicS3Key`.
- **콘텐츠는 토픽을 직접 “소유”하지 않는다.**  
  콘텐츠와 토픽의 연결은 **잡의 `contentId`**로만 간접적으로 이어진다.
- 토픽 플랜 JSON(`TopicPlanResult` 등) **안에 `contentId`가 들어가는 것**은, 생성 시점의 **컨텍스트 스냅샷**에 가깝다(플랜이 콘텐츠 테이블에 FK로 매달린 것은 아님).

### 미연결 잡(placeholder)

- `ADMIN_UNASSIGNED_CONTENT_ID` (`__unassigned__`)는 **카탈로그 콘텐츠가 아닌** 미연결 잡 전용 placeholder다. `createDraftJob`에서 `contentId`를 생략하면 이 값으로 저장된다.
- `attachJobToContent`로 실제 `contentId`를 붙이면 DynamoDB GSI와 S3 시드·브리프·토픽 플랜 JSON이 갱신된다.

### 잡 생성과 토픽 플랜

- 기본값으로 `createDraftJob` 직후에 `runTopicPlan`과 동일한 흐름이 이어져, **한 번의 요청으로 잡 + 토픽 플랜**까지 완료한다. `runTopicPlan: false`로만 시드만 두고 DRAFT로 둘 수 있다.

### 코드·타입에서의 이름 주의

- `AdminJobDto.topicId`는 실질적으로 **`topic_${jobId}` 형태의 잡 파생 ID**에 가깝다. “글로벌 토픽 엔터티의 ID”와 혼동하지 않는다.
- `JobDraftDetailDto` 등에서 **topic plan** 필드가 **TopicSeedDto 타입**으로 노출되는 부분이 있으면, 의미상 혼란(시드 vs 플랜)을 줄이기 위해 플랜 전용 DTO로 분리하는 것이 권장된다.

## 2. 용어 사전 (glossary)

| 용어                      | 의미                                                                                    |
| ------------------------- | --------------------------------------------------------------------------------------- |
| **Content (`contentId`)** | Admin 카태로그의 콘텐츠(채널/라인 단위). 유튜브 연동·라벨 등 발행 맥락.                 |
| **Job (`jobId`)**         | 생성 파이프라인의 한 실행. 시드·플랜·씬 JSON·에셋·업로드 상태의 본체.                   |
| **Topic seed**            | LLM에 넣는 **입력** (제목 아이디어, 길이, 스타일 등). `drafts/.../topic-seed.json` 등.  |
| **Topic plan**            | 시드(및 설정)로부터 나온 **구조화된 플랜** 산출물. `topicS3Key`로 잡에 연결.            |
| **Content brief**         | 잡에 붙는 **기획/제약 스냅샷** (도메인별 필드가 많을 수 있음). seed/plan과 역할이 다름. |
| **Scene JSON**            | 렌더 중립적인 장면 단위 대본/메타. 토픽 플랜 이후 단계.                                 |

---

## 3. 왜 헷갈리는가 (정리 포인트)

- “토픽”이라는 말이 **입력(seed)**, **산출물(plan)**, **잡 파생 ID(`topicId`)**에 동시에 쓰이기 쉽다.
- **콘텐츠**와 **콘텐츠 브리프(Content brief)** 이름이 비슷해, “콘텐츠 = 한 개의 브리프 문서”로 오해하기 쉽다.  
  → **콘텐츠 = 카탈로그 엔터티**, **브리프 = 잡에 붙는 기획 JSON**으로 구분한다.

## 4. 향후 방향(to-be) 옵션: 아이디어 선행 → 콘텐츠 부착

운영 흐름이 아래에 가깝다면, **모델을 바꾸는 것이** UX·언어 모두 정리되기 쉽다.

- 채널(유튜브에 대응하는 **콘텐츠**)을 **나중에** 정한다.
- 아이디어를 **콘텐츠에 종속되지 않은 채로** 키운 뒤, 완성도가 올라가면 **특정 콘텐츠에 붙여** 발행한다.

### 제안하는 개념 (이름은 팀에서 하나로 통일)

| 개념                                      | 역할                                                                                                                                                            |
| ----------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Workspace / Idea / Project** (명칭 택1) | 채널과 무관한 **상위 작업 공간**. 시드→플랜→(선택) 씬까지 같은 파이프라인을 묶을 수 있음.                                                                       |
| **Job**                                   | 지금과 같이 **실행·산출물·상태**의 본체. S3 키는 계속 `jobId` 기준으로 유지 가능.                                                                               |
| **Content**                               | **발행 대상**. Job(또는 상위 Workspace)에 `contentId`를 **나중에 연결(attach)**. 필수 FK가 아닌 **optional → filled** 로 두면 “미연결 작업” 큐를 표현하기 쉽다. |

### 새 상위 엔터티 이름

- 코드에 이미 있는 **`topicId` / `TopicPlan` / `topicS3Key`**와 충돌을 피하려면, 상위 작업 공간에는 **`Idea`**, **`BriefWorkspace`**, **`ProductionProject`** 같이 **다른 어휘**를 쓰는 편이 구현·대화 모두 안전하다.

### 이 모델을 택할 때의 이점

- “콘텐츠에 처음부터 종속”되지 않는 워크플로를 **스키마와 동일한 언어**로 설명할 수 있다.
- **미연결 잡·아이디어 풀** 목록, “발행 대기” 필터 등 UX가 자연스럽다.

### 주의

- **미연결** 엔티티가 늘어나면 **정리·폐기** 정책과 UI가 필요하다.
- **유튜브 메타**(playlist, 기본 공개 등)는 **Content**에, **영상별 제목·설명 초안**은 **Job / Workspace**에 두는 식 **역할 분리**를 문서·타입 모두에서 유지한다.

## 5. 구현 시 체크리스트 (참고)

마이그레이션 없이 문서만 맞출 때:

- [ ] 용어 사전을 팀 기준어로 승인한다.
- [ ] `topicId` 필드의 의미를 주석·OpenAPI/GraphQL description에 명시한다.
- [ ] `topicPlan`과 `topicSeed`의 **DTO 분리** 여부를 결정한다.

아이디어 선행 모델로 갈 때:

- [ ] `contentId`를 Job(또는 상위 엔터티)에서 **optional**로 두고, attach mutation/시점 규칙을 정한다.
- [x] Admin 네비: 전역 “토픽 관리” 메뉴는 제거. `/topics`는 `/jobs`로 리다이렉트. 토픽은 **잡 생성·잡 상세**에서 다룬다.
