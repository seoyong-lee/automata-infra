# Admin 클라이언트 수정 계획 (운영 콘솔 · 후보 채택형 워크벤치)

**Cursor·에이전트 전달용 통합본(도메인 재정의·목록/상세·백엔드 우선순위까지 한 파일):** [`admin-improvement-direction-cursor-handoff.md`](./admin-improvement-direction-cursor-handoff.md)

본 문서는 Admin Web(`apps/admin-web`)의 **정보 구조·도메인 단위·화면 책임**을 재정의하기 위한 수정 방향이다. 백엔드 실행 모델(Resolver 슬림화, Step Functions, Execution 이력 등)은 [`implementation-overview-external-review.md`](../implementation-overview-external-review.md)와 맞춘다. 채널·콘텐츠 운영의 큰 그림은 [`admin-job-authoring.md`](./admin-job-authoring.md)와 연결되나, 여기서는 **후보/채택/스냅샷/실행 이력**과 **메뉴·탭·상세 IA**를 구체적으로 고정한다.

---

## 1. 문제 진단 (현재 UI가 “꼬여” 보이는 이유)

운영자 입장에서 **관리 단위가 한 화면에 섞여** 있다.

- **콘텐츠**가 채널인지, 작품(한 편)인지, 최종 영상 산출물인지 불명확하다.
- **잡**이 “제작 중인 원본”인지, **실행 이력**인지, **템플릿**인지 구분이 안 된다.
- 토픽·시드·스크립트·씬 JSON이 한 덩어리 편집 흐름에 묶여, **여러 후보를 뽑아 두고 베스트를 채택해 다음 단계로 넘기는 흐름**이 드러나지 않는다.
- 결과적으로 **수동 최적화용 작업 공간**과 **자동 실행 파이프라인**이 같은 화면에서 경쟁한다.

표면적으로는 “파이프라인 관리자”처럼 보이지만, 목표에 가깝게 정의하면 다음 **셋의 합성**이다.

1. **콘텐츠 라인 운영 콘솔** — 어느 채널/라인에서 무엇이 돌아가는지
2. **버전 채택형 제작 워크벤치** — 단계마다 후보·채택·다음 실행
3. **실행 파이프라인 모니터** — 무엇이 실패했고 어디서 막혔는지

---

## 2. 수정 방향의 핵심 원칙

### 2.1 콘텐츠를 최소 3계층으로 분리 (멘탈 모델)

| 계층                      | 의미               | 예                                      |
| ------------------------- | ------------------ | --------------------------------------- |
| **A. 채널 / 콘텐츠 라인** | 운영 단위          | Daily Tarot Shorts, Korean Tale Ambient |
| **B. 제작 아이템**        | 개별 산출물(한 편) | “새벽 안개가…” 1편, 오늘의 타로 1편     |
| **C. 실행 / 런**          | 이력 단위          | topic 생성 run #12, scene 생성 run #5   |

지금은 B와 C가 `Job` 하나에 섞여 있어 화면이 어색하다. UI와 데이터 모델 모두에서 **제작 대상(B)** 과 **실행 이력(C)** 을 분리하는 것이 우선이다.

### 2.2 단계마다 “후보”와 “채택본”을 분리

각 스텝은 구조적으로 다음을 갖는다.

- **후보 목록**
- **현재 채택본**
- **채택본 기준 “다음 단계 실행”**
- **실행 이력**

아이데이션 예: 후보 토픽 N개 → 채택 1개 → 채택 기준 brief/scene 생성 → 이전 생성 이력 조회.

### 2.3 자동화는 “채택본 스냅샷” 기준

다음 단계 자동 생성은 **편집 중인 live 값을 즉시 읽어서** 돌리지 않는다.

1. 사용자가 후보 중 하나를 **채택**한다.
2. 채택 시 해당 단계 값이 **스냅샷**으로 고정된다.
3. 이후 실행은 그 **스냅샷 ID**를 입력으로 한다.

그래야 재현성, 버전 추적, “어떤 입력으로 생성됐는지” 설명이 가능하다.

### 2.4 화면은 “입력 폼”이 아니라 “선택·결정” 중심

운영에서 중요한 것은 값 하나를 타이핑하는 것이 아니라:

- 후보 A/B/C 생성
- 비교·평가
- 하나 채택
- 채택본 기준으로 다음 단계 실행
- 이전 단계 채택은 유지한 채 **일부 단계만 재실행**

이 흐름이 각 화면의 1차 액션이 되어야 한다.

## 3. 권장 도메인 모델 (UI·API 정렬용 초안)

백엔드 명칭은 기존 `Content` / `Job`과 점진적으로 맞출 수 있으나, **제품 언어**와 **화면 단위**는 아래 개념에 수렴시키는 것을 권장한다.

| 개념                | 역할               | 주요 필드 예                                                                                                       |
| ------------------- | ------------------ | ------------------------------------------------------------------------------------------------------------------ |
| **Channel**         | 운영 라인          | id, name, platform, format preset, default language, default style preset                                          |
| **Content Item**    | 개별 제작물(한 편) | id, channelId, title, status, target duration, publish target, current approved version 참조                       |
| **Stage Candidate** | 단계별 후보        | id, contentItemId, stageType(ideation, brief, scene, voice, …), versionNo, sourceRunId, payload, score, isApproved |
| **Pipeline Run**    | 실행 이력          | id, contentItemId, stageType, inputSnapshotId, status, startedAt, completedAt, logs, cost, triggeredBy             |
| **Asset**           | 생성 산출물        | 유형(image/audio/render/…), sourceCandidateId, sourceRunId, 승인 여부                                              |

이 분리가 있어야 **수동 최적화**와 **자동 파이프라인**이 같은 제품 안에서 공존한다.

## 4. 메뉴 구조 수정안

### 4.1 기존 (시스템 중심에 가까운 예)

- 대시보드
- 콘텐츠
- 잡
- 작업 현황
- 설정

### 4.2 수정안 (운영자 관점)

- **대시보드**
- **채널** (기존 “콘텐츠”를 채널 관리 의미로 명확화)
- **제작 아이템** (기존 “잡” 대신 — 한 편의 제작 단위)
- **실행 현황** (돌아가는 런·병목·실패 요약)
- **리뷰 큐** (검수·실패 조치)
- **설정**

**“잡”이라는 레이블은 UI에서 줄이거나 대체**한다. 운영자가 필요한 질문은 “어디 소속(채널) / 무엇을 만들고 있는가(제작 아이템) / 무엇이 돌고 있는가(실행) / 무엇을 승인해야 하는가(리뷰)”이다.

## 5. 화면별 수정안

### 5.1 대시보드

숫자 카드만 두지 말고 **운영 액션 보드**로 쓴다.

- **상단 KPI**: 진행 중 제작 아이템, 검수 대기, 실패 실행, 오늘 발행 완료, 에셋 백로그 등
- **중단**: 채널별 상태, 병목 단계, 최근 실패 실행, 승인 대기
- **하단 “바로 처리할 일”**: 검수 N건, 이미지 실패 M건, 업로드 실패 K건 등 클릭 시 해당 큐로 이동

### 5.2 콘텐츠 → 채널

- 목록 컬럼 예: 채널명, 포맷, 기본 길이, 최근 발행일, 활성 제작 아이템 수, 실패/검수 대기 수
- 채널 상세 탭 예: **개요 / 제작 아이템 / 템플릿·프리셋 / 성과 / 설정**

### 5.3 잡 → 제작 아이템

- 목록 컬럼 예: 상태, 제목, 채널, 현재 단계, 채택 버전 여부, 마지막 실행 시각, 검수 상태, 액션
- 상태 예: 아이데이션 중, 씬 확정 대기, 에셋 생성 중, 검수 대기, 업로드 대기, 발행 완료, 실패
- 필터 예: 채널, 상태, 현재 단계, 검수 상태, **미채택 후보 존재 여부**

### 5.4 제작 아이템 상세 — 탭 재구성

**기존 예:** 토픽·시드 / 스크립트 / 이미지 / 영상 / 업로드

**권장:**

| 탭              | 역할                                                                                  |
| --------------- | ------------------------------------------------------------------------------------- |
| **개요**        | 상태, 채널, 타겟 길이, 채택 라인 요약, 마지막 성공 렌더, 검수, 단계 진행 바, 주요 CTA |
| **아이데이션**  | 후보 카드·채택·채택 배너·이 후보 기준 다음 단계                                       |
| **씬 설계**     | brief / scene을 **한 묶음** — 버전 목록 + 구조화 에디터 + Inspector (JSON raw는 고급) |
| **에셋**        | 이미지·음성·BGM·클립·자막을 **씬 단위 행**으로 통합                                   |
| **렌더/업로드** | 프리뷰·최종 렌더·메타·예약·업로드 상태·재업로드 (렌더와 발행은 한 탭, 상태는 분리)    |
| **실행 이력**   | 런 ID, 단계, 입력 버전, 상태, 소요, 비용 추정, 실행자 — 상세에서 스냅샷·로그·재시도   |

**이유:** 토픽·시드와 “스크립트”를 나누는 것만으로는 부족하고, 실제 중심은 **씬 설계**(토픽·brief·scene JSON이 한 단계 흐름으로 연결됨)다.

## 6. 상세 화면 내부 패턴

### 6.1 개요 탭

- 카드: 현재 상태, 채널, 타겟 길이, 채택 라인, 마지막 성공 렌더, 검수
- 액션: 새 후보 생성, 채택본 기준 재생성, 검수 요청, 최종 발행
- 하단: 단계 진행 바(아이데이션 → 씬 설계 → 이미지 → 음성 → 렌더 → 리뷰 → 업로드)

### 6.2 아이데이션 탭

- **좌측**: 후보 리스트(제목, 훅, 스타일, 길이, 출처, 점수/메모)
- **우측**: 선택 후보 상세, 편집, **채택**, “이 후보 기준 brief 생성”, “비슷한 후보 추가 생성”
- **상단 배너**: 현재 채택본, 채택 시점·채택자·버전 번호

목적은 입력이 아니라 **채택**이다.

### 6.3 씬 설계 탭

- **상단**: 채택 아이데이션 / 채택 brief / 채택 scene 버전 요약
- **좌측**: 단계·버전 목록(Brief v1, Scene v3 Approved 등)
- **가운데**: 씬 카드 리스트(narration, subtitle, image/motion prompt, sfx, duration 등)
- **우측 Inspector**: 길이 합계, 톤/금지어 검사, 자동 검증, **이 버전 채택**
- **하단**: 후보 scene N개 생성, 복제, 선택 씬만 재생성, 채택본 기준 에셋 생성 시작
- **JSON**: 기본은 숨기고 “고급 보기”로만 노출

### 6.4 에셋 탭

- 서브탭: 이미지 / 음성 / BGM·SFX / 영상 클립 / 자막
- **씬 단위 표**가 기본: 씬별 채택 에셋, 후보 수, 마지막 생성 시각, 재생성, 수동 업로드

### 6.5 렌더/업로드 탭

- 렌더: 프리뷰·최종·채택·로그
- 업로드: 제목·설명·태그·썸네일·예약·상태·재업로드

### 6.6 실행 이력 탭

- 표: 실행 ID, 단계, 입력 버전, 상태, 소요, 비용 추정, 실행자, 시각
- 패널: input snapshot, output 요약, 로그, 실패 사유, 재시도

## 7. 리뷰 큐 (기존 “작업 현황” 재정의)

평평한 “작업 현황” 대신 **리뷰 큐 + 장애 큐**를 분리한다.

- **검수 대기**: scene / image / preview / upload 등 승인 대기
- **실패/조치 필요**: 이미지·TTS·렌더·업로드 실패 등

카드 단위로 들어가면 해당 제작 아이템·단계로 딥링크한다.

---

## 8. 대표 UX 플로우 (문서·기획 검증용)

### 8.1 수동 탐색형

1. 제작 아이템 생성
2. 아이데이션 후보 여러 개 생성
3. 1개 채택
4. 채택본 기준 scene 후보 생성
5. 일부 수정 후 scene 채택
6. 채택 scene 기준 이미지·음성 자동 생성
7. 실패한 씬만 재생성
8. 프리뷰 승인 → 최종 렌더·업로드

### 8.2 베스트 버전 재사용형

1. 이전 제작 아이템에서 성과 좋았던 approved 라인 선택
2. “같은 라인으로 새 아이템” 생성
3. 제목·시드만 교체
4. scene 재생성 → 에셋 자동 → 검수 후 발행

두 흐름이 **같은 화면 구조**에서 보여야 한다.

## 9. 즉시 적용 가능한 수정 (우선순위)

| 순서 | 내용                                                                                                                                |
| ---- | ----------------------------------------------------------------------------------------------------------------------------------- |
| 1    | 네비: `콘텐츠` → **채널**, `잡` → **제작 아이템**, `작업 현황` → **리뷰 큐** 또는 **운영 현황** (문구·라우트는 점진적 마이그레이션) |
| 2    | 제작 아이템 상세 탭을 **개요 / 아이데이션 / 씬 설계 / 에셋 / 렌더·업로드 / 실행 이력** 으로 재편                                    |
| 3    | Scene JSON **기본 뷰**를 textarea가 아닌 **구조화 뷰**(씬 카드)로 전환, raw는 고급                                                  |
| 4    | 각 주요 단계 화면에 **후보 목록 / 채택본 / 실행 이력** 3영역 패턴 도입                                                              |
| 5    | 상단 CTA를 **저장 단일**에서 **후보 생성 · 채택 · 채택본 기준 다음 단계 · 부분 재실행** 중심으로 재배치                             |

## 10. 한 줄 결론

UI가 어색한 1차 원인은 시각 디자인이 아니라 **도메인 단위가 한 화면에 섞인 것**이다.

**콘텐츠 제작 대상(제작 아이템) · 단계별 후보·채택 · 실행 이력**을 분리하고, UI를 **폼 편집기**가 아니라 **후보 채택형 제작 워크벤치**로 옮기면, 스텝별 수동 최적화·베스트 버전 채택·채택 스냅샷 기반 자동 생성·부분 재실행이 같은 구조 안에서 공존할 수 있다.
